# Cache
## Official Documentation
- [Caching](https://docs.spring.io/spring-boot/docs/current/reference/htmlsingle/#caching)
## 핵심 개념
> Spring Boot는 `@Cacheable`, `@CacheEvict`, `@CachePut` 등 선언적 캐시 추상화를 제공하며, 내부적으로 `CacheManager` 인터페이스를 통해 캐시 구현체(Caffeine, Redis, EhCache 등)와 연동된다. `spring-boot-starter-cache` 의존성만 추가하면 자동 설정이 동작하며, 별도 CacheManager 빈이 없으면 기본 ConcurrentMap 기반 캐시를 사용한다.
>
> 캐시 추상화의 핵심 흐름은 다음과 같다: ① 메서드 호출 시 `CacheInterceptor`(AOP 프록시)가 가로챈다 -> ② `CacheManager`에 등록된 `Cache`에서 키 기반으로 값을 조회한다 -> ③ 캐시에 값이 있으면 메서드를 실행하지 않고 캐시된 값을 반환한다 -> ④ 캐시에 값이 없으면 실제 메서드를 실행하고 그 결과를 캐시에 저장한 뒤 반환한다. 캐시 키는 기본적으로 모든 파라미터를 조합하여 생성되며, SpEL을 통해 커스터마이징할 수 있다.
>
> Spring Boot 3.4.x에서는 `spring.cache.caffeine.spec` 프로퍼티로 Caffeine 사양 문자열을 직접 지정할 수 있어, 별도 `CacheManager` 빈 없이도 `application.yml`만으로 로컬 캐시 정책을 구성할 수 있다.
## 학습 목표
- `@Cacheable`로 메서드 결과를 캐시하고 캐시 키를 커스터마이징할 수 있다.
- `@CacheEvict`로 캐시 무효화 전략(beforeInvocation, allEntries)을 이해한다.
- `@CachePut`으로 캐시 갱신과 메서드 실행을 함께 처리한다.
- `CacheManager` 추상화를 이해하고 Caffeine / Redis 등으로 전환할 수 있다.
- `@Caching`으로 여러 캐시 어노테이션을 조합할 수 있다.
- `application.yml`을 통해 `spring.cache.type` 및 `spring.cache.cache-names`를 설정할 수 있다.
## 예제 코드
```java
@Service
public class UserService {

    private final UserRepository userRepository; // ← 생성자 주입으로 Repository 의존성 확보

    public UserService(UserRepository userRepository) { // ← 생성자 주입 (권장 방식)
        this.userRepository = userRepository;
    }

    @Cacheable(value = "users", key = "#id") // ← 캐시 영역명 "users", 키는 파라미터 id
    public User findById(Long id) {
        simulateSlowService(); // ← DB 조회 등 무거운 작업을 시뮬레이션
        return new User(id, "User" + id);
    }

    @CachePut(value = "users", key = "#user.id") // ← 메서드는 항상 실행되고 결과로 캐시 갱신
    public User update(User user) {
        return userRepository.save(user); // ← DB 저장 후 반환값이 캐시에 덮어쓰기 됨
    }

    @CacheEvict(value = "users", key = "#id") // ← 해당 키의 캐시 항목만 삭제
    public void delete(Long id) {
        userRepository.deleteById(id); // ← DB에서 삭제 후 캐시 무효화
    }

    @Caching(evict = { // ← 여러 캐시 작업을 하나로 묶을 때 사용
        @CacheEvict("users"), // ← "users" 캐시 전체 삭제
        @CacheEvict(value = "userDetails", key = "#user.id") // ← "userDetails"의 특정 키만 삭제
    })
    public void clearAll(User user) {
        // ← 메서드 본문은 비워도 캐시 무효화 어드바이스는 실행됨
    }

    private void simulateSlowService() {
        try { Thread.sleep(3000); } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
    }
}
```
```java
@Configuration
@EnableCaching // ← Spring Boot 3.4.x에서는 자동 설정으로 생략 가능하지만 명시적 선언 권장
public class CacheConfig {

    @Bean
    public CacheManager cacheManager() {
        CaffeineCacheManager cacheManager = new CaffeineCacheManager("users", "userDetails"); // ← 관리할 캐시 이름 목록
        cacheManager.setCaffeine(Caffeine.newBuilder()
                .expireAfterWrite(10, TimeUnit.MINUTES) // ← 쓰기 후 10분 경과 시 만료
                .maximumSize(100)); // ← 캐시 항목 최대 100개, 초과 시 LRU 방식으로 축출
        return cacheManager;
    }
}
```
## 주요 패턴
- Cache-Aside: 애플리케이션이 직접 캐시를 제어하며 `@Cacheable`로 읽고 `@CacheEvict`로 무효화하는 전통적 패턴
- Read-Through / Write-Through: `CacheManager` 구현체가 데이터 소스와 자동 동기화(Spring Cache는 기본적으로 이 모델을 부분 지원)
- 계층형 캐시(Layered Cache): L1(로컬 Caffeine) + L2(분산 Redis)로 구성하여 성능과 확장성을 모두 확보
## 주의사항
- 프록시 기반 AOP이므로 동일 클래스 내부 호출(self-invocation) 시 캐시 어드바이스가 적용되지 않는다.
- `@Cacheable` 메서드의 반환 타입이 `Optional`인 경우 null 캐싱 이슈에 주의해야 한다.
- 기본 `SimpleCacheManager`는 프로덕션에 부적합하므로 반드시 `spring.cache.type`을 명시적으로 설정한다.
- Redis를 사용할 경우 직렬화 방식(Jackson2JsonRedisSerializer 등)을 명시적으로 구성하지 않으면 기본 JDK 직렬화로 인한 문제가 발생할 수 있다.
- **입문자 실수 1**: `spring-boot-starter-cache` 의존성만 추가하고 `spring.cache.type`을 설정하지 않은 채 프로덕션에 배포하면, 기본 ConcurrentMap 캐시가 사용되어 인스턴스 간 캐시 불일치나 메모리 누수가 발생할 수 있다.
- **입문자 실수 2**: `@Cacheable`을 `void` 반환 타입 메서드에 붙이면 캐시에 `null`만 저장되어 의도한 캐싱 효과를 얻을 수 없다. 캐싱이 필요한 메서드는 반드시 캐시할 대상을 반환해야 한다.
