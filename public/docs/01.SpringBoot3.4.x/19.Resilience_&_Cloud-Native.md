# Resilience & Cloud-Native
## Official Documentation
- [Resilience4j Integration (Circuit Breaker, Rate Limiter, Retry)](https://docs.spring.io/spring-boot/docs/current/reference/htmlsingle/#resilience4j.circuitbreaker)
## 핵심 개념
> Spring Boot는 Resilience4j와의 통합을 통해 Circuit Breaker, Rate Limiter, Retry, Bulkhead, Time Limiter와 같은 탄력성 패턴을 선언적 또는 프로그래밍 방식으로 적용할 수 있다. `application.yml` 설정만으로 Circuit Breaker의 임계값, 슬라이딩 윈도우, 폴백 메서드를 구성할 수 있어 마이크로서비스의 장애 전파를 효과적으로 방지한다.
>
> Resilience4j가 제공하는 6가지 핵심 패턴: ① **Circuit Breaker(서킷 브레이커)** — 호출 실패율이 임계값을 초과하면 회로를 OPEN하여 추가 호출을 차단하고, 일정 시간 후 HALF_OPEN 상태에서 소수의 호출을 허용하여 복구 여부를 판단한다. 상태 전이는 CLOSED(정상) → OPEN(차단) → HALF_OPEN(테스트) → CLOSED(복구) 순서로 진행된다. ② **Retry(재시도)** — 일시적인 네트워크 오류나 타임아웃에 대해 지정된 횟수만큼 재시도한다. ③ **Rate Limiter(속도 제한)** — 단위 시간당 최대 호출 횟수를 제한하여 과도한 트래픽으로부터 서비스를 보호한다. ④ **Bulkhead(격벽)** — 리소스를 격리된 그룹으로 분할하여 한 영역의 과부하가 다른 영역으로 전파되는 것을 방지한다. ⑤ **Time Limiter(시간 제한)** — 비동기 작업의 최대 실행 시간을 제한한다. ⑥ **Retry + Circuit Breaker 조합** — Retry를 Circuit Breaker보다 안쪽에 배치하여, 재시도가 모두 실패한 후에만 서킷 브레이커가 실패로 기록하도록 구성한다.
>
> Spring Boot 3.4.x에서는 `resilience4j.circuitbreaker.instances.<name>.sliding-window-type` 속성으로 `COUNT_BASED`(기본값)와 `TIME_BASED` 슬라이딩 윈도우 방식을 선택할 수 있다.
## 학습 목표
- Circuit Breaker 패턴의 상태(CLOSED, OPEN, HALF_OPEN) 이해하기
- `@CircuitBreaker` 어노테이션으로 폴백 메서드 연결하기
- `@Retry`로 일시적 장애에 대한 재시도 정책 구성하기
- Rate Limiter, Bulkhead로 리소스 격리와 트래픽 제어하기
- Actuator를 통해 Circuit Breaker 상태 모니터링하기
## 예제 코드
```java
// Circuit Breaker + Retry 서비스
@Service
public class ExternalApiService {

    private final RestClient restClient; // ← Spring Boot 3.2+ 권장 HTTP 클라이언트 (RestTemplate은 deprecated)

    public ExternalApiService(RestClient.Builder builder) {
        this.restClient = builder.build(); // ← RestClient 빌더 주입
    }

    @CircuitBreaker(name = "inventoryService", fallbackMethod = "inventoryFallback") // ← 서킷 브레이커: 실패율 초과 시 회로 OPEN
    @Retry(name = "inventoryService") // ← Retry: 일시적 오류 발생 시 최대 3회 재시도 (CircuitBreaker보다 먼저 실행됨)
    public String getInventory(String productId) {
        return restClient.get()
            .uri("http://inventory-service/api/inventory/{productId}", productId) // ← URI 템플릿 변수 바인딩
            .retrieve() // ← 응답 본문 추출 준비
            .body(String.class); // ← 응답을 String으로 변환
    }

    public String inventoryFallback(String productId, Throwable t) { // ← 폴백 메서드: 원본 파라미터 + Throwable
        return "{\"stock\": \"UNAVAILABLE\", \"reason\": \"" + t.getMessage() + "\"}"; // ← 장애 시 기본 응답 반환
    }
}
```

```yaml
# application.yml - Resilience4j 설정
resilience4j:
  circuitbreaker:
    instances:
      inventoryService:
        sliding-window-size: 10 # ← 최근 10회 호출 기준으로 실패율 계산
        minimum-number-of-calls: 5 # ← 최소 5회 호출이 쌓여야 실패율 평가 시작
        failure-rate-threshold: 50 # ← 실패율 50% 초과 시 회로 OPEN
        wait-duration-in-open-state: 10s # ← OPEN 상태 유지 시간 (10초 후 HALF_OPEN 전환)
        permitted-number-of-calls-in-half-open-state: 3 # ← HALF_OPEN 시 허용할 호출 수
  retry:
    instances:
      inventoryService:
        max-attempts: 3 # ← 최대 3회 시도 (최초 1회 + 재시도 2회)
        wait-duration: 500ms # ← 재시도 간 대기 시간
  ratelimiter:
    instances:
      apiRateLimiter:
        limit-for-period: 10 # ← 한 주기당 최대 10회 호출 허용
        limit-refresh-period: 1s # ← 1초마다 호출 한도 리셋 (초당 10회 제한)
  timelimiter:
    instances:
      inventoryService:
        timeout-duration: 3s # ← 3초 초과 시 TimeoutException 발생
```
## 주요 패턴
- Fallback Method: 원본 메서드와 동일한 시그니처 + 마지막 파라미터로 `Throwable` 추가
- Chain of Resilience: `@CircuitBreaker` + `@Retry` + `@RateLimiter` 조합
- Event Publishing: `CircuitBreakerEvent`를 구독하여 상태 변경 감지
- Centralized Config: `resilience4j.configs`로 공통 설정을 정의하고 인스턴스에서 재사용
## 주의사항
- Circuit Breaker와 Retry를 함께 사용할 때 Retry가 Circuit Breaker보다 안쪽(먼저 실행)에 배치되어야 한다
- `fallbackMethod`는 반드시 동일한 클래스의 `public` 메서드여야 한다
- Rate Limiter의 `limit-for-period`는 `limit-refresh-period`와 함께 동작하므로 주기 설정이 일관되어야 한다
- Actuator `/health` 엔드포인트에 Circuit Breaker 상태가 포함되려면 `management.health.circuitbreakers.enabled=true` 설정 필요
- **입문자 실수 1**: `fallbackMethod`의 반환 타입과 파라미터 시그니처가 원본 메서드와 다르면 런타임에 `NoSuchMethodException`이 발생한다. 폴백 메서드는 원본 메서드의 모든 파라미터 + 마지막 `Throwable`(선택)을 정확히 일치시켜야 한다.
- **입문자 실수 2**: `minimumNumberOfCalls`를 설정하지 않으면 기본값 100이 적용되어, 소량의 호출로는 서킷 브레이커가 절대 OPEN되지 않는다. 테스트 환경에서는 이 값을 낮게 설정하지 않으면 서킷 브레이커 동작을 확인할 수 없다.
