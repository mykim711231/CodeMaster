# Testing
## Official Documentation
- [Testing Spring Boot Applications](https://docs.spring.io/spring-boot/docs/current/reference/htmlsingle/#testing.boot-application)
## 핵심 개념
> Spring Boot는 `@SpringBootTest`를 통해 통합 테스트 환경을 자동 구성하며, `@WebMvcTest`, `@DataJpaTest`, `@JsonTest` 등 슬라이스 테스트로 특정 레이어만 격리하여 빠르게 테스트할 수 있다. MockMvc는 서블릿 컨테이너 없이 웹 레이어를 테스트하고, `@MockBean` / `@SpyBean`으로 의존성을 교체할 수 있다. Testcontainers를 활용하면 실제 데이터베이스나 메시지 브로커를 도커 컨테이너로 실행하여 프로덕션과 유사한 환경에서 통합 테스트를 수행할 수 있다.
>
> Spring Boot 테스트 전략은 테스트 피라미드에 따라 세 단계로 나뉜다: ① **단위 테스트(Unit)** — Mockito로 의존성을 모킹하여 서비스/유틸리티 클래스의 순수 로직을 검증하며, Spring 컨텍스트 없이 빠르게 실행된다. ② **슬라이스 테스트(Slice)** — `@WebMvcTest`(컨트롤러), `@DataJpaTest`(리포지토리), `@JsonTest`(직렬화) 등 필요한 레이어만 로드하여 컨텍스트 로딩 시간을 최소화한다. ③ **통합 테스트(Integration)** — `@SpringBootTest` + `Testcontainers`로 실제 인프라를 띄워 프로덕션 환경과 가장 유사한 조건에서 E2E 시나리오를 검증한다.
>
> Spring Boot 3.4.x에서는 `@SpringBootTest`의 `useMainMethod` 속성을 통해 메인 메서드 실행 방식을 제어할 수 있으며, `@ServiceConnection` 어노테이션으로 Testcontainers 컨테이너의 연결 정보를 자동 주입할 수 있다.
## 학습 목표
- `@SpringBootTest`로 전체 ApplicationContext를 로드하여 통합 테스트를 작성할 수 있다.
- `@WebMvcTest`로 컨트롤러 레이어만 로드하고 MockMvc로 HTTP 요청을 검증할 수 있다.
- `@DataJpaTest`로 JPA 리포지토리를 테스트하고 자동 롤백을 활용할 수 있다.
- `@MockBean`으로 서비스 의존성을 모킹하고 `@SpyBean`으로 부분 모킹할 수 있다.
- `TestRestTemplate`을 사용하여 실제 내장 서버에서 엔드투엔드 테스트를 수행할 수 있다.
- Testcontainers로 PostgreSQL, Redis, Kafka 등 실제 인프라와 통합 테스트를 구성할 수 있다.
- `@TestConfiguration`으로 테스트 전용 설정 빈을 추가할 수 있다.
- `@SpringBootTest(webEnvironment = WebEnvironment.RANDOM_PORT)`로 실제 포트로 구동하여 E2E 테스트를 수행할 수 있다.
## 예제 코드
```java
@WebMvcTest(UserController.class) // ← UserController 레이어만 로드 (서비스, 리포지토리 제외)
class UserControllerTest {

    @Autowired
    private MockMvc mockMvc; // ← 서블릿 컨테이너 없이 HTTP 요청 시뮬레이션

    @MockBean
    private UserService userService; // ← 실제 UserService 대신 Mock 객체를 컨텍스트에 등록

    @Test
    void shouldReturnUser() throws Exception {
        User user = new User(1L, "Alice");
        given(userService.findById(1L)).willReturn(user); // ← Mock 동작 정의: findById(1L) 호출 시 user 반환

        mockMvc.perform(get("/users/1") // ← GET /users/1 요청 수행
                        .accept(MediaType.APPLICATION_JSON)) // ← JSON 응답 기대
                .andExpect(status().isOk()) // ← HTTP 200 검증
                .andExpect(jsonPath("$.name").value("Alice")); // ← 응답 JSON의 name 필드가 "Alice"인지 검증
    }

    @Test
    void shouldReturn404WhenUserNotFound() throws Exception {
        given(userService.findById(99L)).willThrow(new UserNotFoundException()); // ← 예외 발생 시나리오 모킹

        mockMvc.perform(get("/users/99"))
                .andExpect(status().isNotFound()); // ← HTTP 404 검증
    }
}
```
```java
@DataJpaTest // ← JPA 관련 빈만 로드하고 각 테스트 후 자동 롤백
class UserRepositoryTest {

    @Autowired
    private TestEntityManager entityManager; // ← 테스트용 엔터티 매니저 (EntityManager 래퍼)

    @Autowired
    private UserRepository userRepository; // ← 실제 JPA 리포지토리 주입

    @Test
    void shouldFindByEmail() {
        User user = new User("alice@example.com", "Alice");
        entityManager.persistAndFlush(user); // ← DB에 저장 + 즉시 flush (ID 부여됨)

        Optional<User> found = userRepository.findByEmail("alice@example.com"); // ← 쿼리 메서드 실행

        assertThat(found).isPresent(); // ← AssertJ: 조회 결과가 존재하는지
        assertThat(found.get().getName()).isEqualTo("Alice"); // ← 이름이 일치하는지
    }
}
```
```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT) // ← 실제 내장 서버를 랜덤 포트로 구동
@Testcontainers // ← Testcontainers 확장 활성화 (컨테이너 생명주기 관리)
class UserIntegrationTest {

    @Container
    static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:16-alpine"); // ← static으로 선언하여 테스트 클래스 간 컨테이너 재사용

    @DynamicPropertySource
    static void configureProperties(DynamicPropertyRegistry registry) { // ← 컨테이너 정보를 Spring 프로퍼티로 동적 등록
        registry.add("spring.datasource.url", postgres::getJdbcUrl); // ← 실제 PostgreSQL JDBC URL 주입
        registry.add("spring.datasource.username", postgres::getUsername);
        registry.add("spring.datasource.password", postgres::getPassword);
    }

    @Autowired
    private TestRestTemplate restTemplate; // ← 실제 HTTP 요청을 보내는 테스트용 RestTemplate

    @Test
    void shouldCreateAndRetrieveUser() {
        User newUser = new User("bob@example.com", "Bob");
        ResponseEntity<User> createResponse = restTemplate.postForEntity("/users", newUser, User.class); // ← POST 요청으로 사용자 생성
        assertThat(createResponse.getStatusCode()).isEqualTo(HttpStatus.CREATED); // ← 201 Created 검증

        ResponseEntity<User> getResponse = restTemplate.getForEntity(
                "/users/{id}", User.class, createResponse.getBody().getId()); // ← 생성된 ID로 GET 조회
        assertThat(getResponse.getBody().getName()).isEqualTo("Bob"); // ← 저장된 이름과 일치하는지 검증
    }
}
```
## 주요 패턴
- Slice Testing: `@WebMvcTest`, `@DataJpaTest`, `@JsonTest`로 전체 컨텍스트 대신 특정 슬라이스만 빠르게 로드
- BDDMockito: `given().willReturn()` 스타일로 Given-When-Then 구조의 가독성 높은 테스트 코드 작성
- Test Fixture 재사용: `@TestConfiguration` + `@Import`로 여러 테스트 클래스에서 공통 목(mock) 빈과 픽스처 공유
- Testcontainers Singleton: `@Container`를 static 필드로 선언하여 테스트 클래스 간 컨테이너를 재사용
## 주의사항
- `@SpringBootTest`는 전체 컨텍스트를 로드하므로 테스트 실행이 느리다. 가능하면 슬라이스 테스트를 우선 사용한다.
- `@MockBean`은 ApplicationContext를 오염시켜 컨텍스트 캐싱을 무력화할 수 있으므로, 동일한 구성의 테스트 클래스끼리 그룹화해야 한다.
- `@DataJpaTest`는 기본적으로 H2 인메모리 DB를 사용하므로, 프로덕션 DB(PostgreSQL 등)와 SQL 문법 차이로 인해 테스트가 통과되더라도 런타임 오류가 발생할 수 있다.
- TestRestTemplate은 서버를 실제로 구동하므로 `webEnvironment = RANDOM_PORT` 또는 `DEFINED_PORT`를 설정해야 한다.
- **입문자 실수 1**: `@DataJpaTest`에서 `@MockBean`으로 Repository를 모킹하면 JPA 자동 설정이 의도대로 동작하지 않는다. `@DataJpaTest`는 실제 인메모리 DB를 사용하는 것이 목적이므로, Repository 모킹이 필요하면 `@SpringBootTest` + `@MockBean`을 사용해야 한다.
- **입문자 실수 2**: `@WebMvcTest(UserController.class)`에 컨트롤러 클래스를 지정하지 않으면 모든 컨트롤러가 로드되어 슬라이스 테스트의 이점이 사라진다. 테스트 대상 컨트롤러를 반드시 명시해야 한다.
