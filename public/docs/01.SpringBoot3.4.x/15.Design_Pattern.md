# Design Pattern
## Official Documentation
- [Design Patterns (Java)](https://refactoring.guru/design-patterns/java)
## 핵심 개념
> 디자인 패턴은 소프트웨어 설계에서 반복적으로 발생하는 문제에 대한 검증된 해결책을 제공하며, 생성(Creational), 구조(Structural), 행위(Behavioral)의 세 가지 범주로 분류된다. Spring 프레임워크 자체도 싱글톤(Bean Scope), 프록시(AOP), 템플릿 메서드(JdbcTemplate), 전략(ResourceResolver) 등 다수의 GoF 패턴을 내부적으로 활용한다. 패턴을 이해하면 Spring Boot의 동작 원리를 더 깊이 파악할 수 있고, 유지보수성과 확장성이 높은 코드를 작성할 수 있다.
>
> Spring Boot 애플리케이션에서 특히 자주 사용되는 패턴들을 실제 기술 요소와 연결하면 다음과 같다: ① **싱글톤(Singleton)** — Spring IoC 컨테이너의 기본 빈 스코프로, `@Service`, `@Repository`, `@Controller`가 붙은 클래스는 기본적으로 애플리케이션 전역에서 하나의 인스턴스만 생성되어 공유된다. ② **프록시(Proxy)** — `@Transactional`, `@Cacheable`, `@Async`, `@Retry` 등 AOP 기반 어노테이션은 모두 프록시 객체를 통해 원본 빈을 감싸 부가 기능을 주입한다. ③ **템플릿 메서드(Template Method)** — `JdbcTemplate`, `RestTemplate`, `RedisTemplate`은 데이터 접근의 반복적인 보일러플레이트(연결 관리, 예외 변환, 리소스 해제)를 숨기고 비즈니스 로직만 콜백으로 제공하게 한다. ④ **전략(Strategy)** — 인터페이스를 정의하고 여러 `@Component` 구현체를 주입받아, 런타임 조건에 따라 알고리즘을 교체한다(예: 결제 수단, 할인 정책 등). ⑤ **옵저버(Observer)** — `ApplicationEvent`와 `@EventListener`로 서비스 간 느슨한 결합을 유지하며 이벤트 기반 워크플로우를 구성한다.
>
> 패턴 학습 시 주의할 점은, "패턴을 위한 코드"가 아니라 "문제 해결을 위한 패턴"이어야 한다는 것이다. 패턴을 적용하기 전에 반드시 당면한 문제가 해당 패턴이 해결하려는 문제와 일치하는지 확인해야 한다.
## 학습 목표
- 싱글톤 패턴과 Spring의 기본 Bean Scope(싱글톤)의 관계를 이해할 수 있다.
- 팩토리 메서드 / 추상 팩토리 패턴으로 객체 생성 로직을 캡슐화할 수 있다.
- 빌더 패턴으로 복잡한 객체를 단계적으로 구성하고 Lombok `@Builder`와 연계할 수 있다.
- 전략 패턴으로 런타임에 알고리즘을 교체하고 `@Service` + 인터페이스로 Spring DI와 결합할 수 있다.
- 옵저버 패턴을 Spring의 `ApplicationEvent` / `@EventListener`로 구현할 수 있다.
- 템플릿 메서드 패턴이 `JdbcTemplate`, `RestTemplate`, `KafkaTemplate` 등에서 어떻게 적용되었는지 이해할 수 있다.
- 데코레이터 / 프록시 패턴과 Spring AOP(@Transactional, @Cacheable)의 관계를 설명할 수 있다.
- 어댑터 패턴으로 외부 API 응답을 도메인 모델로 변환하는 레이어(Port-Adapter)를 구현할 수 있다.
## 예제 코드
```java
// 전략 패턴 - 결제 방식에 따라 전략 교체
public interface PaymentStrategy { // ← 전략 인터페이스: 모든 결제 방식이 구현해야 할 계약
    PayResult pay(PayRequest request);
}

@Component("CREDIT_CARD") // ← 빈 이름을 "CREDIT_CARD"로 지정 (Map 키로 사용)
class CreditCardPayment implements PaymentStrategy {
    public PayResult pay(PayRequest request) {
        return new PayResult("CARD", request.amount(), "approved"); // ← 신용카드 결제 로직
    }
}

@Component("KAKAO_PAY") // ← 빈 이름을 "KAKAO_PAY"로 지정
class KakaoPayPayment implements PaymentStrategy {
    public PayResult pay(PayRequest request) {
        return new PayResult("KAKAO", request.amount(), "approved"); // ← 카카오페이 결제 로직
    }
}

@Service
class PaymentService {
    private final Map<String, PaymentStrategy> strategies; // ← 결제 수단명 -> 전략 객체 매핑

    public PaymentService(List<PaymentStrategy> strategyList) { // ← Spring이 모든 PaymentStrategy 구현체를 List로 주입
        this.strategies = strategyList.stream()
                .collect(Collectors.toMap(
                        s -> s.getClass().getAnnotation(Component.class).value(), // ← @Component의 value()를 Map 키로 추출
                        Function.identity())); // ← 전략 객체 자체를 Map 값으로 사용
    }

    public PayResult process(PayRequest request) {
        PaymentStrategy strategy = strategies.get(request.method()); // ← 요청의 결제 수단명으로 전략 조회
        if (strategy == null) throw new IllegalArgumentException("Unknown method"); // ← 미지원 결제 수단 방어
        return strategy.pay(request); // ← 선택된 전략 실행
    }
}
```
```java
// 옵저버 패턴 - Spring Events
@Data // ← final 필드만 있으므로 @Data는 @Getter + @RequiredArgsConstructor + @ToString + @EqualsAndHashCode 생성
public class OrderPlacedEvent {
    private final Long orderId; // ← 이벤트에 포함할 데이터: 주문 ID
    private final String userId; // ← 이벤트에 포함할 데이터: 사용자 ID
}

@Component
class EmailNotificationListener {
    @EventListener // ← OrderPlacedEvent 발행 시 이 메서드가 자동 호출됨
    @Async // ← 비동기 실행: 이메일 발송이 주문 처리 트랜잭션을 지연시키지 않도록
    public void handle(OrderPlacedEvent event) {
        emailService.send(event.getUserId(), "Your order #" + event.getOrderId() + " is placed");
    }
}

@Component
class InventoryUpdateListener {
    @EventListener
    @Order(1) // ← 여러 리스너 중 이 리스너를 가장 먼저 실행 (숫자가 낮을수록 높은 우선순위)
    public void handle(OrderPlacedEvent event) {
        inventoryService.deduct(event.getOrderId()); // ← 재고 차감 (이메일 발송보다 먼저 처리)
    }
}

@Service
class OrderService {
    private final ApplicationEventPublisher publisher; // ← 이벤트 발행기 (Spring 제공)

    @Transactional
    public Order placeOrder(OrderRequest request) {
        Order order = orderRepository.save(new Order(request)); // ← 주문 저장 (DB 커밋)
        publisher.publishEvent(new OrderPlacedEvent(order.getId(), request.getUserId())); // ← 이벤트 발행 (기본적으로 동기)
        return order;
    }
}
```
```java
// 빌더 패턴 - Lombok @Builder + 도메인 객체
@Builder // ← 빌더 클래스 및 builder() 정적 메서드 자동 생성
@Getter // ← 모든 필드에 대한 getter 자동 생성
public class User {
    private final Long id; // ← final: 생성 이후 불변 (빌더로만 설정 가능)
    private final String name;
    private final String email;
    private final Integer age;
    private final LocalDateTime createdAt;

    public static User createDefault(String name, String email) { // ← 편의용 정적 팩토리 메서드
        return User.builder() // ← Lombok이 생성한 builder() 호출
                .name(name) // ← 필드별로 값 설정
                .email(email)
                .createdAt(LocalDateTime.now()) // ← 기본값 주입 (현재 시간)
                .build(); // ← 최종 객체 생성 (id, age는 null 허용)
    }
}
```
## 주요 패턴
- 싱글톤: Spring IoC 컨테이너의 기본 Bean Scope — 하나의 빈 인스턴스가 컨테이너 내에서 공유됨
- 팩토리 메서드: Spring의 `@Bean` 메서드, `FactoryBean<T>` 인터페이스로 객체 생성 로직을 추상화
- 전략: 인터페이스 + 여러 `@Component` 구현체를 주입받아 런타임에 알고리즘 분기 제거
- 옵저버: `ApplicationEvent` + `@EventListener`로 모듈 간 느슨한 결합을 유지하며 이벤트 기반 통신
- 템플릿 메서드: `JdbcTemplate`, `RedisTemplate`, `RestTemplate` 등에서 반복적인 보일러플레이트(연결, 해제, 예외 처리)를 숨기고 핵심 로직만 노출
- 프록시: Spring AOP — `@Transactional`, `@Cacheable`, `@Async` 등이 프록시 객체를 통해 횡단 관심사를 주입
- 데코레이터: BeanPostProcessor, `@Decorated` 어노테이션 등으로 기존 빈에 부가 기능을 동적으로 추가
- 어댑터: 외부 API DTO ↔ 도메인 모델 변환 계층을 분리하여 외부 의존성 변화가 도메인으로 전파되는 것을 방지
## 주의사항
- 싱글톤 빈에 상태(멤버 변수)를 저장하면 동시성 문제가 발생하므로, 상태는 메서드 파라미터나 프로토타입 빈으로 분리해야 한다.
- 전략 패턴의 구현체를 `List<>`로 주입받을 때 주입 순서가 보장되지 않으므로, 순서가 필요하면 `@Order`나 명시적 Map 키 전략을 사용한다.
- Spring AOP는 프록시 기반이므로 `private` 메서드에는 `@Transactional`, `@Async` 등이 적용되지 않는다.
- 디자인 패턴을 과도하게 적용하면 오히려 코드 복잡도가 증가하므로, 실제로 해결해야 할 문제가 있을 때만 적용한다.
- `ApplicationEvent`는 기본적으로 동기 실행되므로, 시간이 오래 걸리는 리스너는 `@Async`로 비동기 처리해야 한다.
- **입문자 실수 1**: `@EventListener` 메서드 내에서 예외가 발생하면 현재 트랜잭션이 롤백되지 않는다. 이벤트 리스너는 기본적으로 발행자(publisher) 트랜잭션과 분리되어 실행되므로, 리스너 실패 시 발행자 로직이 되돌려지길 기대해서는 안 된다. 트랜잭션 범위를 제어하려면 `@TransactionalEventListener`를 사용해야 한다.
- **입문자 실수 2**: Lombok `@Builder`를 클래스에 적용할 때 모든 필드가 `final`이면 컴파일 오류가 발생하지 않지만, `@NoArgsConstructor`와 함께 사용하면 컴파일 오류가 발생한다. `@Builder`를 사용할 때는 어떤 생성자를 기반으로 빌더가 동작하는지 이해해야 한다.
