# Data
## Official Documentation
- [Data Access (SQL, NoSQL)](https://docs.spring.io/spring-boot/docs/current/reference/htmlsingle/#data)
## 핵심 개념
> Spring Boot Data 모듈은 관계형 DB(JDBC, JPA)와 NoSQL(MongoDB, Redis, Cassandra, Neo4j) 등 다양한 데이터 저장소에 대해 일관된 접근 추상화를 제공한다. 자동 설정을 통해 커넥션 풀(HikariCP), 트랜잭션 관리, Repository 인터페이스 생성을 지원하므로 데이터 액세스 계층을 빠르게 구축할 수 있다.
>
> Spring Data의 핵심 아키텍처는 다음 계층으로 구성된다: ① **Repository 추상화** — `CrudRepository` → `PagingAndSortingRepository` → `JpaRepository` / `MongoRepository`로 이어지는 인터페이스 계층 구조로, 저장소 유형과 무관하게 동일한 CRUD 패턴을 제공한다. ② **쿼리 메서드(Query Derivation)** — 메서드 이름을 분석하여 `findByNameAndAgeGreaterThan(String, int)` 같은 메서드 시그니처만으로 JPQL/쿼리를 자동 생성한다. ③ **`@Query`** — 복잡한 JPQL, SQL, 네이티브 쿼리를 직접 작성할 수 있으며, `nativeQuery = true`로 데이터베이스 고유 SQL도 실행 가능하다. ④ **자동 설정(Auto-Configuration)** — 클래스패스에 H2, PostgreSQL, MongoDB 드라이버가 존재하면 `DataSourceAutoConfiguration`이 해당 데이터소스를 자동 구성하며, HikariCP가 기본 커넥션 풀로 설정된다. ⑤ **스키마 관리** — `ddl-auto` 옵션으로 개발 중 스키마 자동 생성을 지원하며, Flyway 또는 Liquibase를 통해 프로덕션 스키마 버전 관리를 수행한다.
>
> Spring Boot 3.4.x에서 JDBC Client(`JdbcClient`)가 추가되어, `JdbcTemplate`보다 더 간결한 Fluent API로 SQL 실행이 가능하다. 또한 R2DBC를 통한 리액티브 DB 접근도 정식 지원된다.
## 학습 목표
- Spring Data JPA로 `JpaRepository` 인터페이스 기반 CRUD 구현하기
- `@Entity`, `@Table`, `@Column`으로 엔터티 매핑하기
- MongoDB, Redis, Cassandra 스타터로 NoSQL 데이터소스 구성하기
- `DataSource`와 HikariCP 커넥션 풀 설정 이해하기
- 쿼리 메서드, `@Query`, QueryDSL로 복잡한 조회 구현하기
## 예제 코드
```java
// JPA 엔터티
@Entity // ← JPA가 관리하는 엔터티 클래스임을 선언
@Table(name = "products") // ← 매핑될 DB 테이블명 지정 (생략 시 클래스명을 소문자로 변환)
public class Product {
    @Id // ← 기본 키(PK) 필드
    @GeneratedValue(strategy = GenerationType.IDENTITY) // ← DB의 AUTO_INCREMENT로 PK 생성
    private Long id;

    @Column(nullable = false, length = 100) // ← NOT NULL 제약 & 최대 길이 100
    private String name;

    @Column(nullable = false)
    private BigDecimal price;
    // getters, setters ← 실제 코드에서는 Lombok @Getter @Setter 또는 수동 구현 필요
}

// JPA Repository
public interface ProductRepository extends JpaRepository<Product, Long> { // ← <엔터티, PK타입>
    List<Product> findByPriceBetween(BigDecimal min, BigDecimal max); // ← 쿼리 메서드: 메서드명으로 JPQL 자동 생성

    @Query("SELECT p FROM Product p WHERE p.name LIKE CONCAT('%', :keyword, '%')") // ← JPQL 직접 작성 (LIKE 검색)
    List<Product> searchByName(@Param("keyword") String keyword); // ← @Param으로 명명된 파라미터 바인딩
}
```

```yaml
# application.yml - JPA + H2 설정
spring:
  datasource:
    url: jdbc:h2:mem:testdb # ← H2 인메모리 DB (테스트/개발용)
    driver-class-name: org.h2.Driver
    hikari:
      maximum-pool-size: 10 # ← HikariCP 커넥션 풀 최대 크기
  jpa:
    hibernate:
      ddl-auto: update # ← 엔터티 기반으로 테이블 자동 생성/갱신 (프로덕션 금지!)
    show-sql: true # ← 실행되는 SQL을 콘솔에 출력 (개발 디버깅용)
    properties:
      hibernate:
        format_sql: true # ← 출력되는 SQL을 보기 좋게 포맷팅
```

```java
// MongoDB Repository
@Document(collection = "products") // ← MongoDB 컬렉션명 지정 (생략 시 클래스명을 소문자로 변환)
public class ProductDocument {
    @Id // ← MongoDB 문서의 _id 필드 (String 타입이면 ObjectId 자동 생성)
    private String id;
    private String name;
    private BigDecimal price;
}

public interface ProductMongoRepository extends MongoRepository<ProductDocument, String> { // ← MongoRepository<문서타입, ID타입>
    List<ProductDocument> findByPriceGreaterThan(BigDecimal price); // ← 쿼리 메서드: price > ? 조건 자동 생성
}
```
## 주요 패턴
- Repository 추상화: 동일한 인터페이스 패턴으로 JPA/MongoDB/Cassandra 모두 지원
- 자동 설정(DataSourceAutoConfiguration): 클래스패스 기반 데이터소스 자동 구성
- DTO Projection: 엔터티 일부 필드만 조회하는 인터페이스/클래스 프로젝션
- Flyway/Liquibase: 데이터베이스 마이그레이션 자동 실행
## 주의사항
- `spring.jpa.hibernate.ddl-auto=update`는 프로덕션에서 사용하지 말 것 (데이터 손실 위험)
- `@Transactional`은 public 메서드에만 적용되며 AOP 프록시 내부 호출은 무시된다
- MongoDB는 ID 필드에 `@Id`를 반드시 지정해야 하며, String 타입이면 ObjectId 자동 생성
- Redis를 세션 저장소로 사용할 때 `@EnableRedisHttpSession` 설정 누락에 주의
- **입문자 실수 1**: `JpaRepository`의 `getById()`(Spring Data JPA 2.x)와 `getReferenceById()`(3.x)를 혼동하여 사용하는 실수. 전자는 deprecated 되었고, 후자는 프록시 참조만 반환하므로 실제 필드 접근 시 LazyInitializationException이 발생할 수 있다. ID로 엔터티를 즉시 조회하려면 `findById()`를 사용해야 한다.
- **입문자 실수 2**: `ddl-auto=update`로 운영 DB에 배포한 후 엔터티 필드를 삭제하거나 이름을 변경했을 때, Hibernate가 기존 컬럼을 자동으로 DROP 하지 않아 불일치가 누적되는 문제. 운영에서는 반드시 Flyway/Liquibase로 명시적 마이그레이션을 관리해야 한다.
