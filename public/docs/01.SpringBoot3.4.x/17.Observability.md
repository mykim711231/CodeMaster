# Observability
## Official Documentation
- [Logging, Tracing, Metrics](https://docs.spring.io/spring-boot/docs/current/reference/htmlsingle/#logging)
## 핵심 개념
> Spring Boot는 Logback을 기본 로깅 프레임워크로 사용하며, Micrometer를 통해 다양한 모니터링 백엔드에 메트릭을 전송할 수 있다. Spring Boot Actuator와 OpenTelemetry 연동으로 분산 트레이싱까지 통합 관측 가능성(Observability)을 구축할 수 있으며, `application.yml`만으로 로그 레벨, 형식, 출력 대상을 세밀하게 제어할 수 있다.
>
> 관측 가능성(Observability)은 모니터링을 넘어선 개념으로, 시스템의 내부 상태를 외부 출력만으로 추론할 수 있는 능력을 의미하며 **로깅(Logs)**, **메트릭(Metrics)**, **트레이싱(Tracing)** 세 가지 신호(signal)로 구성된다: ① **로깅** — 이벤트 기반의 불연속적인 텍스트 기록으로, `logging.level`, `logging.pattern`, `logging.file` 속성으로 제어하며 ELK 스택(Elasticsearch + Logstash + Kibana)과 연동 시 JSON 구조화 로깅이 권장된다. ② **메트릭** — 시계열 수치 데이터(JVM 힙 사용량, HTTP 요청 수, DB 커넥션 풀 사용률 등)로, Micrometer가 Prometheus, Datadog, CloudWatch 등 백엔드에 대한 Facade 역할을 하며 `@Timed`, `@Counted`로 비즈니스 메트릭을 추가할 수 있다. ③ **트레이싱** — 하나의 요청이 여러 마이크로서비스를 통과하는 경로를 추적하며, Spring Boot 3.x부터 Micrometer Tracing + Brave(또는 OpenTelemetry) 조합으로 `traceId`/`spanId`를 자동 전파한다. Zipkin, Grafana Tempo, Jaeger 등으로 시각화할 수 있다.
## 학습 목표
- `application.yml`에서 로그 레벨, 패턴, 파일 출력 설정하기
- Logback-spring.xml로 고급 로깅 커스터마이징하기
- Actuator 엔드포인트(/health, /metrics, /info) 활성화 및 이해하기
- Micrometer로 Prometheus, Grafana 연동하기
- OpenTelemetry와 Zipkin으로 분산 트레이싱 구성하기
## 예제 코드
```yaml
# application.yml - 로깅 및 Actuator 설정
logging:
  level:
    root: INFO # ← 전체 로그 레벨은 INFO
    com.example: DEBUG # ← 특정 패키지는 DEBUG로 상세 로깅
  pattern:
    console: "%d{yyyy-MM-dd HH:mm:ss} [%thread] %-5level %logger{36} - %msg%n" # ← 콘솔 출력 형식 지정
  file:
    name: logs/application.log # ← 파일 출력 경로 (기본은 출력하지 않음)

management:
  endpoints:
    web:
      exposure:
        include: health,info,metrics,prometheus # ← 노출할 Actuator 엔드포인트 명시적 지정
  metrics:
    export:
      prometheus:
        enabled: true # ← Prometheus scraping을 위한 /actuator/prometheus 활성화
  tracing:
    sampling:
      probability: 1.0 # ← 개발 환경: 모든 요청을 트레이싱 (운영에서는 0.1 이하)
  zipkin:
    tracing:
      endpoint: http://localhost:9411/api/v2/spans # ← Zipkin 서버 엔드포인트 (Brave 사용)
```

```java
// Controller에서 커스텀 메트릭 등록
import io.micrometer.core.instrument.MeterRegistry;
import org.springframework.web.bind.annotation.*;

@RestController
public class OrderController {

    private final MeterRegistry meterRegistry; // ← Micrometer 메트릭 등록소 (생성자 주입)

    public OrderController(MeterRegistry meterRegistry) {
        this.meterRegistry = meterRegistry;
    }

    @GetMapping("/orders/{id}")
    public String getOrder(@PathVariable String id) {
        meterRegistry.counter("orders.requests", "endpoint", "getOrder").increment(); // ← 커스텀 카운터 +1 (Prometheus에서 orders_requests_total로 노출)
        return "Order: " + id;
    }
}
```
## 주요 패턴
- 구조화 로깅(JSON): ELK 스택 연동을 위한 `logstash-logback-encoder` 사용
- SLF4J 파라미터화: `log.info("User {} logged in", userId)`로 성능 최적화
- 로그 그룹핑: `logging.group`으로 패키지별 일괄 로그 레벨 설정
- 스팬/트레이스 전파: HTTP 헤더를 통한 `traceId`/`spanId` 자동 전달
## 주의사항
- 프로덕션에서는 `sampling.probability`를 0.1 이하로 설정해 트레이싱 오버헤드 최소화
- Actuator 엔드포인트는 반드시 Spring Security로 보호하거나 `exposure.exclude`로 제한
- `@Timed` 어노테이션은 AOP 프록시 내부 호출에서는 동작하지 않는다
- OTel 에이전트와 Micrometer 양쪽에서 동일 메트릭을 중복 수집하지 않도록 주의
- **입문자 실수 1**: `management.tracing.sampling.probability=1.0`으로 설정한 채 운영 배포하면, 모든 요청의 트레이스가 수집되어 애플리케이션 성능이 크게 저하되고 트레이싱 백엔드 저장소가 빠르게 포화된다. 운영에서는 `0.1`(10%) 이하로 설정해야 한다.
- **입문자 실수 2**: `logging.file.name`만 설정하고 `logging.logback.rollingpolicy.file-name-pattern` 등 로테이션 정책을 생략하면 로그 파일이 무제한으로 커져 디스크를 가득 채울 수 있다. 파일 로깅 사용 시 반드시 `max-history`와 `max-file-size`를 함께 설정해야 한다.
