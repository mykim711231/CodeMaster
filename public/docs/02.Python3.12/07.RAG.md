# RAG

## Official Documentation
- [LangChain Introduction](https://python.langchain.com/docs/introduction/)

## 핵심 개념
> RAG(Retrieval-Augmented Generation)는 LLM의 응답 생성을 외부 지식 베이스 검색으로 보강하는 패러다임이다. LLM이 학습하지 않은 최신 정보나 비공개 문서에 대해, 먼저 관련 문맥을 검색한 후 이를 프롬프트에 함께 주입하여 사실에 기반한 답변을 생성한다. 문서를 청킹(의미 단위로 분할)한 후 임베딩 모델로 벡터화하여 VectorStore(Chroma, FAISS, Pinecone 등)에 저장하고, 사용자 쿼리와 유사한 청크를 검색해 프롬프트에 주입한다. Retriever로 초기 후보를 얻고 Reranker(Cross-Encoder)로 정밀도를 높이며, Hybrid Search(BM25 + Dense)로 키워드 기반 검색과 의미 기반 검색을 결합해 검색 정확도를 극대화한다.

## 학습 목표
- 문서를 의미 있는 청크로 분할하는 RecursiveCharacterTextSplitter 등 청킹 전략
- 청크를 임베딩 벡터로 변환해 Chroma, FAISS 등 VectorStore에 저장
- 쿼리 임베딩과 유사도 검색으로 관련 청크 Retrieval 파이프라인 구축
- Reranker를 적용해 1차 검색 결과의 관련성 재정렬
- Hybrid Search로 키워드 기반(BM25) + 의미 기반(벡터) 검색 통합
- 검색된 컨텍스트와 사용자 질문을 결합한 최종 프롬프트 조립 및 생성

## 예제 코드
```python
from langchain.text_splitter import RecursiveCharacterTextSplitter  # ← 재귀적 문자 분할기
from langchain_community.vectorstores import Chroma  # ← 경량 로컬 VectorStore
from langchain_community.embeddings import HuggingFaceEmbeddings  # ← HuggingFace 임베딩 모델
from langchain.retrievers import ContextualCompressionRetriever  # ← 압축 검색기 (Reranker 래퍼)
from langchain.retrievers.document_compressors import CrossEncoderReranker  # ← Cross-Encoder 재정렬기
from langchain_community.cross_encoders import HuggingFaceCrossEncoder  # ← HuggingFace Cross-Encoder

# 1. 문서 청킹 (의미 단위로 분할)
text_splitter = RecursiveCharacterTextSplitter(  # ← 구분자 우선순위에 따라 재귀 분할
    chunk_size=500,  # ← 각 청크의 최대 문자 수 (토큰 기준 약 125~150 토큰)
    chunk_overlap=50,  # ← 인접 청크 간 중복 문자 수 (문맥 연속성 유지)
    separators=["\n\n", "\n", ".", " ", ""]  # ← 분할 우선순위: 문단 > 줄바꿈 > 문장 > 공백 > 강제분할
)
chunks = text_splitter.split_documents(documents)  # ← Document 객체 리스트를 청크로 분할

# 2. 임베딩 및 VectorStore 저장
embeddings = HuggingFaceEmbeddings(model_name="BAAI/bge-base-en-v1.5")  # ← BGE 임베딩 모델 로드 (768차원)
vectorstore = Chroma.from_documents(  # ← 문서 -> 임베딩 -> Chroma DB에 저장
    chunks,  # ← 분할된 청크 리스트
    embeddings,  # ← 임베딩 모델
    persist_directory="./chroma_db"  # ← 디스크 저장 경로 (재사용 가능)
)

# 3. Retriever로 검색 (기본 유사도 검색)
retriever = vectorstore.as_retriever(search_kwargs={"k": 10})  # ← 상위 10개 문서 검색
docs = retriever.invoke("What is RAG?")  # ← 쿼리 실행 -> 관련 문서 리스트 반환

# 4. Reranker로 재정렬 (1차 검색 결과를 정밀 재평가)
reranker_model = HuggingFaceCrossEncoder(model_name="BAAI/bge-reranker-base")  # ← Cross-Encoder 로드
compressor = CrossEncoderReranker(model=reranker_model, top_n=3)  # ← 상위 3개만 남기고 압축
compression_retriever = ContextualCompressionRetriever(  # ← Retriever + Compressor 결합
    base_compressor=compressor, base_retriever=retriever
)
reranked_docs = compression_retriever.invoke("What is RAG?")  # ← 검색 -> Reranker 순위 재조정

# 5. Hybrid Search (BM25 키워드 검색 + Dense 벡터 검색 앙상블)
from langchain.retrievers.ensemble import EnsembleRetriever  # ← 앙상블 검색기
from langchain_community.retrievers import BM25Retriever  # ← BM25 키워드 기반 검색기

bm25_retriever = BM25Retriever.from_documents(chunks)  # ← 청크로 BM25 인덱스 구축
bm25_retriever.k = 5  # ← BM25에서 반환할 문서 개수

ensemble_retriever = EnsembleRetriever(  # ← 두 검색 결과를 가중 평균으로 결합
    retrievers=[bm25_retriever, vectorstore.as_retriever(k=5)],  # ← BM25 + Dense Retriever
    weights=[0.4, 0.6]  # ← 가중치: BM25 40%, Dense 60% (의미 검색에 더 높은 가중치)
)
hybrid_docs = ensemble_retriever.invoke("What is RAG?")  # ← 하이브리드 검색 실행
```

## 주요 패턴
- 청킹: 시맨틱(구분자 기반) vs 고정 길이 vs 문장 단위 분할, 오버랩으로 문맥 유지
- 불변/가변 필터: `metadata` 기반 필터로 날짜, 출처 등 특정 문서만 검색
- Reranker: Bi-Encoder(임베딩) -> Cross-Encoder(재정렬) 2단계 파이프라인
- Hybrid Search: BM25(키워드) + Dense Retriever(벡터) -> Ensemble/Fusion
- 컨텍스트 윈도우: 검색된 청크 수(`top_k`)와 모델의 최대 컨텍스트 길이 간 균형
- VectorStore 선택: Chroma(경량/로컬), FAISS(고속), Pinecone/Weaviate(관리형)
- 프롬프트 템플릿: 질문 + 검색 컨텍스트 + 지시사항을 구조화된 프롬프트로 조립

## 주의사항
- 청크 크기가 너무 작으면 문맥 손실, 너무 크면 관련 없는 정보 포함 -> 256~1024 토큰이 일반적
- 임베딩 모델과 검색 쿼리에 적합한지 확인 (비대칭 질문-문서 검색 시 instruction prefix 필요)
- VectorStore 인덱스를 사전 빌드하고 재사용하지 않으면 초기화 시간이 오래 걸림
- Reranker는 Cross-Encoder 기반이므로 모든 쿼리-문서 쌍을 비교해 속도 비용이 큼
- Hybrid Search에서 BM25는 형태소 분석 기반이므로 한국어 등 언어에 맞는 토크나이저 설정 필요
- 프롬프트에 컨텍스트를 주입할 때 모델 최대 토큰을 초과하지 않도록 `total_tokens ≤ max_tokens` 검증
- `Chroma.from_documents()`에 `persist_directory`를 지정하지 않으면 인메모리 모드로 동작하여 프로세스 종료 시 데이터가 소멸된다.
- 임베딩 모델과 Reranker 모델은 서로 다른 목적(Bi-Encoder vs Cross-Encoder)에 최적화되어 있으므로, 동일 모델을 두 용도로 사용하지 않는다.
