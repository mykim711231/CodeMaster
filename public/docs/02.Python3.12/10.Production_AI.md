## 핵심 개념
> FastAPI는 고성능 비동기 Python 웹 프레임워크로 `@app.get` 데코레이터와 `async def`로 엔드포인트를 정의한다. Starlette 기반으로 동작하며, OpenAPI/Swagger 문서를 자동 생성한다. Pydantic의 `BaseModel`과 `Field`로 요청/응답 데이터 검증·직렬화를 수행하며, Pydantic v2에서는 Rust 기반 `pydantic-core`로 5~50배 빠른 검증이 가능하다. SQLAlchemy 2.0+에서는 `DeclarativeBase`와 `Mapped[]` 타입 어노테이션으로 ORM 모델을 정의하며, `select()` 구문 중심의 새로운 쿼리 API를 사용한다. Celery+Redis로 비동기 작업 큐를 구성하고, pytest·mypy·ruff로 테스트·타입 체크·린팅을 통해 프로덕션 품질을 확보한다.
## 학습 목표
- FastAPI로 REST API 엔드포인트 정의 및 `async/await` 비동기 핸들러 구현
- Pydantic BaseModel, Field로 데이터 스키마 정의 및 유효성 검증
- SQLAlchemy ORM으로 CRUD 레포지토리 및 DB 마이그레이션 관리
- Celery + Redis로 장시간 실행 작업을 비동기 Task Queue로 처리
- BackgroundTasks로 간단한 비동기 작업을 FastAPI 내에서 처리
- pytest fixtures, mocking, TestClient로 API 통합 테스트 작성
- mypy로 정적 타입 검사, ruff로 린팅·포매팅을 통한 코드 품질 관리
## 예제 코드
```python
import asyncio
from fastapi import FastAPI, BackgroundTasks, Depends, HTTPException
from pydantic import BaseModel, Field
from sqlalchemy import create_engine, Column, Integer, String
from sqlalchemy.orm import declarative_base, sessionmaker, Session
from celery import Celery

# ← FastAPI 앱 인스턴스 생성 (모든 엔드포인트의 진입점)
app = FastAPI()
# ← Celery 앱: Redis를 메시지 브로커로 사용해 작업을 전달해요
celery_app = Celery("tasks", broker="redis://localhost:6379/0")

# Pydantic 스키마
class ItemCreate(BaseModel):
    # ← Field(...)는 필수값, min_length/max_length로 길이 제한
    name: str = Field(..., min_length=1, max_length=100)
    # ← gt=0: 0보다 큰 값만 허용 (greater than)
    price: float = Field(..., gt=0)

class ItemResponse(ItemCreate):
    # ← 응답용 스키마: ItemCreate를 상속받고 id 필드를 추가
    id: int
    class Config:
        from_attributes = True  # ← ORM 객체에서 직접 변환 가능하도록 설정

# SQLAlchemy 모델
engine = create_engine("sqlite:///./items.db")  # ← SQLite DB 연결 (파일 기반)
Base = declarative_base()  # ← 모든 ORM 모델의 부모 클래스
SessionLocal = sessionmaker(bind=engine)  # ← DB 세션 팩토리

class ItemModel(Base):
    __tablename__ = "items"
    id = Column(Integer, primary_key=True)  # ← 자동 증가 기본키
    name = Column(String(100))
    price = Column(Integer)

Base.metadata.create_all(bind=engine)  # ← 모델을 실제 DB 테이블로 생성

# ← FastAPI 의존성 주입용 함수: 요청마다 새로운 DB 세션을 제공
def get_db():
    db = SessionLocal()
    try:
        yield db  # ← yield로 세션을 전달하고, 응답 완료 후 finally 실행
    finally:
        db.close()

# FastAPI 엔드포인트
# ← @app.get: GET /items/{item_id} 요청을 처리
@app.get("/items/{item_id}", response_model=ItemResponse)
async def get_item(item_id: int, db: Session = Depends(get_db)):
    # ← Depends(get_db): FastAPI가 자동으로 get_db()를 호출해 db 세션 주입
    item = db.query(ItemModel).filter(ItemModel.id == item_id).first()
    if not item:
        raise HTTPException(status_code=404, detail="Item not found")
    return item  # ← response_model에 따라 ItemResponse로 자동 직렬화

# Celery Task
@celery_app.task
def process_item_async(item_id: int):
    # ← Celery Worker가 이 함수를 별도 프로세스에서 실행해요
    import time
    time.sleep(5)  # 실제로는 AI 추론 등 무거운 작업
    return {"status": "completed", "item_id": item_id}

# ← POST /items/process/{item_id}: Celery 작업을 큐에 등록
@app.post("/items/process/{item_id}")
async def trigger_processing(item_id: int):
    task = process_item_async.delay(item_id)  # ← .delay()로 비동기 작업 전송
    return {"task_id": task.id, "status": "queued"}

# BackgroundTasks
# ← POST /items/: 새 아이템 생성 + 배경 작업 등록
@app.post("/items/", response_model=ItemResponse)
async def create_item(
    item: ItemCreate,
    background_tasks: BackgroundTasks,
    db: Session = Depends(get_db)
):
    # ← model_dump()로 Pydantic 모델을 dict로 변환 후 ORM 객체 생성 (Pydantic v2 방식)
    db_item = ItemModel(**item.model_dump())
    db.add(db_item)
    db.commit()
    db.refresh(db_item)  # ← DB에서 생성된 id 등 최신 값 반영
    # ← 응답 전송 "후"에 Celery 작업을 실행 → 사용자 응답 시간 단축
    background_tasks.add_task(process_item_async.delay, db_item.id)
    return db_item
```

## 주요 패턴
- Depends(): 의존성 주입으로 DB 세션, 인증, 권한 등을 엔드포인트에 주입
- Pydantic v2: `model_dump()`, `model_validate()`, `TypeAdapter`로 직렬화 및 검증
- Async/Sync DB: `asyncpg` + `SQLAlchemy[asyncio]`로 비동기 DB 접근
- Alembic: SQLAlchemy 모델 변경을 DB 마이그레이션으로 버전 관리
- Celery Task Group/Chain: `group()`, `chain()`으로 병렬/순차 작업 워크플로우
- Request/Response Header/Path/Query Validation: `Header()`, `Path()`, `Query()` 유효성 검증
- pytest fixture + factory: 테스트 DB 초기화와 Fixture factory로 반복 테스트 효율화
- ruff: `ruff check --fix . && ruff format .`으로 린트 오류 자동 수정 및 포매팅
## 주의사항
- `async def`에서 동기 I/O 호출(`time.sleep()`, blocking ORM) 시 이벤트 루프 블로킹 발생 → `run_in_executor()` 사용
- Celery worker가 Redis 브로커 연결을 유지하지 못하면 작업 손실 → `acks_late=True`로 신뢰성 확보
- Pydantic v1 → v2 마이그레이션 시 `from_orm` → `from_attributes`, `dict()` → `model_dump()`로 변경
- SQLAlchemy 2.0+에서는 `DeclarativeBase`로 모델 정의 방식 변경 → `Mapped[]` 타입 어노테이션 권장
- FastAPI TestClient는 `lifespan` 이벤트와 실제 이벤트 루프를 사용 → @pytest.mark.anyio로 비동기 테스트 분리
- Production 배포 시 `uvicorn` 단독이 아닌 `gunicorn + uvicorn workers`로 프로세스 관리
- `.env` 파일과 `pydantic-settings`로 시크릿/환경변수를 안전하게 관리
- BackgroundTasks는 서버 프로세스 내에서 실행되므로, 서버 재시작 시 작업이 소멸됨 → 영속성이 필요한 작업은 반드시 Celery 사용
- Celery의 `.delay()`는 인자를 직렬화(JSON/Pickle)하므로, ORM 객체 같은 복잡한 객체 대신 id 등 기본 타입만 전달
