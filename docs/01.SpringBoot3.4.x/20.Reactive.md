# Reactive
## Official Documentation
- [WebFlux (Reactive Web)](https://docs.spring.io/spring-boot/docs/current/reference/htmlsingle/#webflux)
## 핵심 개념
> Spring WebFlux는 Reactive Streams 기반의 논블로킹 웹 프레임워크로, Netty 또는 Undertow를 기본 런타임으로 사용한다. `Mono`(0..1)와 `Flux`(0..N) 타입을 통해 비동기 데이터 스트림을 선언적으로 조합하며, 함수형 라우터와 애너테이션 기반 컨트롤러 두 가지 스타일을 모두 지원한다. 적은 스레드로 높은 동시성을 처리할 수 있어 I/O 집약적 워크로드에 적합하다.
>
> 리액티브 프로그래밍의 핵심 개념은 **논블로킹(비동기)** 과 **배압(Backpressure)** 이다: ① **논블로킹 I/O** — 요청 스레드가 I/O 완료를 기다리지 않고 즉시 반환되며, 이벤트 루프(Netty의 EventLoop)가 적은 수의 스레드로 수천 개의 동시 연결을 처리한다. 전통적인 서블릿 기반 모델이 요청당 하나의 스레드를 할당하는 것과 대조적이다. ② **Reactive Streams 사양** — `Publisher`, `Subscriber`, `Subscription`, `Processor` 4가지 인터페이스로 구성된 표준 사양이며, Project Reactor는 이 사양의 구현체로 `Mono<T>`(0~1개 데이터)와 `Flux<T>`(0~N개 데이터)를 제공한다. ③ **Operator 파이프라인** — `map`, `flatMap`, `filter`, `zip`, `merge` 등의 연산자를 체이닝하여 데이터 흐름을 선언적으로 변환하며, 실제 구독(subscribe)이 발생하기 전까지는 아무것도 실행되지 않는 지연 실행(Lazy Execution) 특성을 가진다. ④ **배압(Backpressure)** — 소비자가 생산자의 데이터 생성 속도를 따라가지 못할 때, 소비자가 생산자에게 속도를 조절해 달라고 신호를 보내는 메커니즘이다. `limitRate()`, `onBackpressureBuffer()` 등으로 제어한다.
>
> WebFlux 사용 시 주의할 핵심 원칙은 **"절대 블로킹하지 말 것"** 이다. 핸들러나 서비스 계층에서 `block()`, `blockFirst()`를 호출하거나 JDBC와 같은 블로킹 드라이버를 사용하면, 적은 스레드가 블로킹되어 전체 처리량이 급감한다.
## 학습 목표
- `Mono`와 `Flux`의 생성, 변환(map/flatMap/filter), 구독 이해하기
- 애너테이션 기반 `@RestController` + `Mono/Flux`로 Reactive API 구현하기
- 함수형 라우터(`RouterFunction` + `HandlerFunction`)로 엔드포인트 정의하기
- `WebClient`로 논블로킹 HTTP 클라이언트 구현하기
- R2DBC로 리액티브 데이터베이스 액세스 구현하기
## 예제 코드
```java
// 애너테이션 기반 Reactive Controller
@RestController
@RequestMapping("/users")
public class UserController {

    private final UserRepository userRepository; // ← ReactiveCrudRepository를 구현한 Repository (R2DBC 기반)

    public UserController(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    @GetMapping
    public Flux<User> getAll() { // ← Flux: 0~N개의 User를 논블로킹 스트림으로 반환
        return userRepository.findAll(); // ← DB에서 전체 User를 Flux로 반환 (구독 시점에 쿼리 실행)
    }

    @GetMapping("/{id}")
    public Mono<ResponseEntity<User>> getById(@PathVariable String id) {
        return userRepository.findById(id) // ← Mono<User> 반환 (0 또는 1건)
            .map(ResponseEntity::ok) // ← 데이터 있으면 HTTP 200으로 래핑
            .defaultIfEmpty(ResponseEntity.notFound().build()); // ← 데이터 없으면 HTTP 404
    }

    @PostMapping
    @ResponseStatus(HttpStatus.CREATED)
    public Mono<User> create(@RequestBody User user) {
        return userRepository.save(user); // ← 저장된 User를 Mono로 반환 (응답 본문에 포함)
    }
}
```

```java
// 함수형 라우터
@Configuration
public class RouterConfig {

    @Bean
    public RouterFunction<ServerResponse> userRoutes(UserHandler handler) { // ← 라우터 함수 Bean 등록
        return RouterFunctions.route() // ← RouterFunctions.route()로 빌더 시작
            .GET("/api/users", handler::getAll) // ← GET /api/users → handler.getAll()
            .GET("/api/users/{id}", handler::getById) // ← GET /api/users/{id} → handler.getById()
            .POST("/api/users", handler::create) // ← POST /api/users → handler.create()
            .build(); // ← 최종 RouterFunction 빌드
    }
}

@Component
public class UserHandler {

    private final UserRepository repository;

    public UserHandler(UserRepository repository) {
        this.repository = repository;
    }

    public Mono<ServerResponse> getAll(ServerRequest request) { // ← ServerRequest에서 요청 정보 추출 가능
        return ServerResponse.ok().body(repository.findAll(), User.class); // ← Flux<User>를 응답 본문으로
    }

    public Mono<ServerResponse> getById(ServerRequest request) {
        return repository.findById(request.pathVariable("id")) // ← 경로 변수 {id} 추출
            .flatMap(user -> ServerResponse.ok().bodyValue(user)) // ← Mono<User> → Mono<ServerResponse>
            .switchIfEmpty(ServerResponse.notFound().build()); // ← empty Mono일 때 404
    }

    public Mono<ServerResponse> create(ServerRequest request) {
        return request.bodyToMono(User.class) // ← 요청 본문을 User로 변환
            .flatMap(repository::save) // ← 저장
            .flatMap(user -> ServerResponse.created(URI.create("/api/users/" + user.getId())) // ← 201 Created + Location 헤더
                    .bodyValue(user)); // ← 저장된 User를 응답 본문으로
    }
}
```

```java
// WebClient - 논블로킹 HTTP 호출
@Service
public class UserService {

    private final WebClient webClient; // ← RestTemplate의 리액티브 대체제

    public UserService(WebClient.Builder builder) { // ← WebClient.Builder는 자동 설정 빈으로 제공됨
        this.webClient = builder.baseUrl("http://localhost:8080").build(); // ← 기본 URL 설정 후 빌드
    }

    public Flux<User> getUsers() {
        return webClient.get() // ← GET 요청 시작
            .uri("/api/users") // ← 요청 URI
            .retrieve() // ← 응답 수신 준비
            .bodyToFlux(User.class); // ← 응답 본문을 Flux<User>로 변환 (스트리밍 지원)
    }
}
```
## 주요 패턴
- Operator Chaining: `map` → `flatMap` → `filter` → `switchIfEmpty`의 선언적 파이프라인
- Backpressure: `limitRate()`로 스트림 소비 속도를 조절하여 생산자-소비자 속도 불일치 해결
- Scheduler: `subscribeOn()`/`publishOn()`으로 스레드 전환 제어
- Reactive Repository: R2DBC `ReactiveCrudRepository`로 논블로킹 DB 액세스
## 주의사항
- Blocking API(`block()`, `blockFirst()`)를 WebFlux 핸들러 내에서 호출하면 논블로킹의 이점이 사라진다
- Spring MVC와 WebFlux는 동시에 사용할 수 없으며, WebFlux 스타터를 선택하면 Netty가 기본 서버로 설정된다
- `flatMap`은 순서를 보장하지 않으므로 순서 보장이 필요하면 `flatMapSequential` 또는 `concatMap` 사용
- R2DBC 사용 시 JPA(`@Entity` + Hibernate)는 사용할 수 없고, `@Table` 기반의 Spring Data R2DBC 매핑을 사용해야 한다
- **입문자 실수 1**: WebFlux 의존성을 추가하고도 `spring-boot-starter-web`을 함께 사용하면, Spring MVC가 우선되어 WebFlux가 아닌 Tomcat + 서블릿 기반으로 동작한다. WebFlux를 사용하려면 `spring-boot-starter-web`을 제거하고 `spring-boot-starter-webflux`만 유지해야 한다.
- **입문자 실수 2**: `Mono`/`Flux`를 반환하는 메서드에 `.subscribe()`를 직접 호출하여 구독을 수동으로 시작하는 실수. WebFlux에서는 프레임워크가 자동으로 구독하므로, `.subscribe()`를 호출하면 의도치 않은 중복 실행과 스레드 누수가 발생할 수 있다. 구독은 프레임워크에 맡기고 Operator 체이닝으로 흐름만 정의해야 한다.
