# LLM Serving
## Official Documentation
- [vLLM 공식 문서](https://docs.vllm.ai/en/latest/)
- [Ollama](https://github.com/ollama/ollama)
- [TGI (Text Generation Inference)](https://huggingface.co/docs/text-generation-inference/index)

## 핵심 개념
> LLM 서빙은 대규모 언어 모델을 프로덕션 환경에서 효율적으로 추론하기 위한 기술이다. vLLM은 PagedAttention 알고리즘으로 KV Cache 메모리를 OS의 가상 메모리처럼 페이지 단위로 관리하여 메모리 단편화를 해결하고, 연속 배칭(Continuous Batching)으로 요청이 완료되는 즉시 새 요청을 배치에 추가해 GPU 활용률을 극대화한다. Ollama는 GGUF 포맷의 양자화 모델을 로컬에서 단일 명령어로 실행할 수 있게 해주는 경량 서빙 도구로, llama.cpp를 기반으로 동작한다. TGI(Text Generation Inference)는 Hugging Face의 프로덕션급 서빙 프레임워크이며, SSE(Server-Sent Events)를 통해 토큰 단위 스트리밍을 제공한다. GGUF는 llama.cpp에서 사용하는 양자화 모델 포맷으로 2-bit~8-bit까지 다양한 양자화 레벨을 지원하며, AWQ는 활성화 중요도를 반영한 가중치 양자화로 품질 손실을 최소화한다.

## 학습 목표
- vLLM으로 OpenAI 호환 API 서버 구축 및 배치 추론 설정하기
- Ollama로 GGUF/AWQ 양자화 모델 로컬 실행하기
- SSE 스트리밍을 통한 토큰 단위 실시간 응답 구현하기
- KV Cache 메모리 관리 전략과 PagedAttention 원리 이해하기
- 배치 추론 시 동적 배칭과 연속 배칭의 차이 이해하기

## 예제 코드
```python
# vLLM 서버 실행 (CLI)
# python -m vllm.entrypoints.openai.api_server --model mistralai/Mistral-7B-Instruct-v0.2
```
```python
# vLLM 클라이언트 - OpenAI 호환 API
from openai import OpenAI

# ← vLLM 서버에 OpenAI 호환 API로 연결 (base_url로 vLLM 서버 지정)
client = OpenAI(
    base_url="http://localhost:8000/v1",  # ← vLLM 기본 포트는 8000
    api_key="not-needed",  # ← 로컬 서버이므로 API 키 불필요
)

# ← chat.completions.create: OpenAI 표준 Chat Completion API와 동일한 인터페이스
response = client.chat.completions.create(
    model="mistralai/Mistral-7B-Instruct-v0.2",
    messages=[{"role": "user", "content": "Hello!"}],
    max_tokens=256,
    temperature=0.7,
    stream=True,  # ← 스트리밍 모드: 토큰이 생성될 때마다 즉시 전송
)
# ← 스트리밍 응답: chunk 단위로 순회하며 실시간 출력
for chunk in response:
    if chunk.choices[0].delta.content:  # ← delta.content가 None이 아닐 때만 출력
        print(chunk.choices[0].delta.content, end="", flush=True)
```
```python
# Ollama Python 클라이언트
import ollama

# ← ollama.chat(): 로컬 Ollama 서버에 채팅 요청 (기본 포트 11434)
response = ollama.chat(
    model="llama3",
    messages=[{"role": "user", "content": "Explain quantum computing"}],
    stream=True,  # ← 토큰 단위 스트리밍 응답
)
for chunk in response:
    # ← Ollama 스트리밍 청크는 dict 형태, message.content에 토큰이 담겨요
    print(chunk["message"]["content"], end="", flush=True)
```

```python
# Ollama GGUF 모델 import
# Modelfile 내용:
# FROM ./model.Q4_K_M.gguf     ← 로컬 GGUF 파일 경로

```
```python
# SSE 스트리밍 서버 직접 구현
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import asyncio

app = FastAPI()

# ← async generator: yield로 데이터를 하나씩 생성해 클라이언트에 전송
async def generate_tokens(prompt: str):
    tokens = ["안녕", "하세요", "!", "저는", "AI", "입니다"]
    for token in tokens:
        yield f"data: {token}\n\n"  # ← SSE 형식: "data: 내용\n\n"
        await asyncio.sleep(0.5)    # ← 실제 LLM 토큰 생성 지연 시뮬레이션
    yield "data: [DONE]\n\n"        # ← 스트리밍 종료 신호

# ← StreamingResponse: SSE 스트림을 HTTP 응답으로 전송
@app.get("/stream")
async def stream(prompt: str = "hello"):
    return StreamingResponse(
        generate_tokens(prompt),
        media_type="text/event-stream",  # ← SSE MIME 타입
    )
```

## 주요 패턴
- PagedAttention: KV Cache를 페이지 단위로 관리하여 메모리 단편화 해결
- 연속 배칭(Continuous Batching): 요청별로 다른 길이의 시퀀스를 동적으로 배치
- GGUF 양자화: 4-bit, 8-bit 등 다양한 비트 양자화로 모델 크기 대폭 축소 (e.g. Q4_K_M, Q5_K_M)
- AWQ(Activation-aware Weight Quantization): 활성화 중요도 기반 가중치 양자화로 품질 손실 최소화
- OpenAI 호환 API: vLLM/Ollama를 표준 API로 래핑하여 클라이언트 호환성 확보

## 주의사항
- AWQ 모델은 양자화에 사용된 calibration 데이터와 배포 환경이 크게 다르면 품질 저하
- vLLM은 일부 모델 아키텍처에서 지원되지 않을 수 있음 (호환성 확인 필수)
- SSE 스트리밍 시 네트워크 중단에 대비한 재연결 로직 필요
- GPU 메모리가 부족하면 배치 크기를 줄이거나 양자화 모델을 사용해야 함
- vLLM의 `api_key="not-needed"`는 개발 환경 전용이며, 실제 배포 시 반드시 인증 설정 필요
- Ollama의 GGUF 모델명은 `ollama list`로 확인한 정확한 이름을 사용해야 하며, Hub에서 자동 pull되지 않은 모델은 `ollama create`로 먼저 등록
